Build notes — what each piece does

  • Two roles only (v1): Adult (parent, grandparent, carer — equal access, full app) and Kid (Kids Hub focused, can chat with Zaeli, limited family-wide editing). No granular permissions yet.
  • Invite trigger: Our Family screen → "+ Invite" CTA card OR tap an existing kid member → role is pre-set if known. Settings → Family also surfaces the same flow.
  • iOS share sheet: after "Send invite", standard system share opens. Sender picks Messages / Mail / WhatsApp / Copy. Recipient = any device — they get the link.
  • Invite link: zaeli.app/i/<token>. v1 = local mock token (no backend cross-device). Real token validation lands with backend pass.
  • Invite token (backend pass): Supabase row — family_id, role, invited_email, invited_name, token, status, expires_at. One-time-use, 7-day expiry.
  • SMS copy varies by role: Adult = "Rich invited you to join our family on Zaeli." Kid = "Mum set up Zaeli for our family — your own space, jobs, games, Tutor."
  • Adult onboarding: 3 steps — confirm name + password / their own brief times / their preferences (skip-able). Then full 11-stop tour. ~90 sec + tour.
  • Kid onboarding: 2 steps — pick avatar + 4-digit PIN / quick capability intro. Lands in Kids Hub (NOT family chat). PIN instead of password (kid-friendly).
  • Pending state: Family screen shows mint-tinted member with "PENDING" tag + Copy/Resend/Revoke chips. Multiple invites can be active.
  • Inviter heads-up: when invitee accepts, Zaeli pushes a tinted message in chat (mint for adults, lavender for kids) with a follow-up offer.
  • Trust the link: no approval flow. Anyone with the link joins. Safer flows (token expiry, single-use) come with backend pass.
  • Kid permissions: sees own Hub + chat + Tutor. Cannot edit family calendar/meals/shopping. Can suggest jobs/rewards (existing approval flow stays).
